koajs 日志和错误处理最佳实践

发布时间：2018 年 07 月 14 日 最近更新于：2018 年 07 月 18 日 本文共 3,294 字，读完约 13 分钟 
这篇文章总结了我这一年的 kaojs 项目中对日志和错误处理的思考和实践(附详细的实现思路)，希望可以给大家提供一些思考。

[TOC]

日志要具有的功能

日志作为后端发现问题、排查问题、定位问题必不可少的工具，一直在后端架构中占据着重要的地位。

相对于前端，后端所处的环境有以下几个特点：

1，代码运行在服务器，服务提供方需要对代码的运行状态进行监控；
2，发生错误后，不应该(有的时候也很难)要求调用方再调你的接口，配合你的排查；
3，请求数量多且复杂，特别在高并发的场景下。

根据以上特点，日志要具有以下功能：

1，当程序发生错误时，可以及时通知维护者，并提供相关错误信息(message、堆栈等)。
2，日志要有自我描述能力，包括请求的详细参数、经过的函数、返回的结果，从而可以很快定位问题；最不济也可以根据日志完整的mock出对方的请求，重现问题；
3，每个请求分配一个requestId；并且可以根据条件(requestId、时间、ip、用户、接口、请求参数、返回的 code 等等)进行筛选。这样就不会被无关请求的日志干扰。

类似的代码很多。因为在调用接口出错的时候，要排查问题，是需要知道请求参数和返回结果的。但日志过多很块就会挤爆日志文件。而且这些日志平常并没有记录的必要，只有在排查问题的时候才会用到。那么有没有办法做到，只有在出问题的时候才打印这部分日志？

bunyan内置了一个RingBuffer对象就可以解决这个问题。写入到这个对象的日志会存储在内存中，可以对写入的日志条数进行限制(如100条)，超过限制会删除最早写入的日志。

bunyan对日志级别进行分类，从下到上分别为log.trace, log.debug, log.info, log.warn, log.error, 和 log.fatal。比如上述信息一般使用debug打印。

我们可以根据日志的等级决定输出的位置。默认只打印info级别以上日志，而把debug以上级别的日志都打印一份到RingBuffer中。在需要时将RingBuffer中的日志(最近100条)打印出来。如下：

再进一步：开发环境日志(服务端和浏览器端)

开发的过程中，经常需要用console.log打印出日志进行调试，但开发完成后，这些语句就需要被删掉。

如果你也是这样，我推荐你使用 debug 。它允许为每个debug设置命名空间，从而对日志进行开关。

具体使用方式如下：

1 . 引入debug并用命名空间对其进行初始化

2 . 像使用 console.log 一样使用 debug，如

3 . 通过设置全局变量，能对上述打印日志的语句进行开关。 nodejs 中通过环境变量 DEBUG 对debug进行开关，浏览器中通过 localStorage.debug 对debug进行开关。

  • 将 DEBUG 或 localStorage.debug 设置为 *，则所有 debug 的日志都会打印出来；
  • 将 DEBUG 或 localStorage.debug 设置为project:dir1:*, 则只会激活命名空间以project:dir1开头的debug；
  • 将 DEBUG 或 localStorage.debug 设置为严格匹配，比如project:dir1:subDir1:filename1，则只会激活以该字符串作为命名空间的debug。

4 . 很多开源的库都用 debug 对日志进行开关，所以在设置环境变量的时候需要小心，如果都设置为 ，那么可能会多出很多无关的日志信息。推荐在 node 端，在开发环境将 DEBUG 设置为 `:-bable, 在生产环境将 DEBUG 设置为‘’，即关闭debug。在浏览器端，在开发环境将localStorage.debug设置为*`。在生产环境用户不会设置该变量，所以在控制台不会打印任何日志。但你进行调试时，可以通过该变量对日志进行开关，从而更好的调试。

5 . 性能问题。
debug 的引入并不会对性能造成影响。debug 是在用命名空间初始化的时候，就对该初始化的 debug 是否被激活进行了判断，如果没有被激活，那么会用一个变量存储该结果，在调用初始化好的 debug 时，会简单的 return 回去。所以，不是每次调用 debug 时都实时进行判断，这样对性能的影响微乎其微。不过这样也会有个问题，就是每次改变了环境变量DEBUG,就需要重启服务才能生效。在浏览器端，则需要刷新页面。
