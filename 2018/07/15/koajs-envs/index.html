nodejs 多环境配置最佳实践

发布时间：2018 年 07 月 15 日 最近更新于：2018 年 07 月 18 日 本文共 2,071 字，读完约 8 分钟

在后端项目中，随着业务的增长和部署环境的增多，配置会越来越多。怎么在配置增多的情况下，还保持配置之间实现最大的复用，并且清晰明了，是一项非常重要的能力。

在最近一个 nodejs 项目中，我完成的后端需要部署在五个环境: 开发环境、测试环境、公有云环境、私有云环境1、私有云2环境。这五个环境网络相互独立，底层依赖方地址不同。但其他逻辑一致。怎么样把这部分不同点抽出为配置，并对配置进行管理，总结了一套实践方案。

[TOC]

需求整理

我把后端对配置的需求整理如下：

  1. 账号、密码等不能被 git 跟踪

    如果将账号密码等用 git 跟踪，并保存到内部 gitlab 上，很容易导致账号密码等被泄露。包括数据库的账号密码，这是极大的安全漏洞。

  2. 不涉密的配置信息可以被 git 跟踪，但需要根据环境的不同对下层暴露不同的配置。从而在业务代码上层打平不同环境之间的不同点。

  3. 配置需要尽可能的复用

    Don’t Repeat Yourself (DRY)
    不要重复你自己，包括配置。

还有两点需要注意：

  1. 要实现在不同文件中 var config = require('config') 取到配置的效果，需要先将配置初始化。可以在入口文件中先 require一次。
  2. 配置文件是根据环境变量 HOSTNAME 进行寻找的，而环境变量 HOSTNAME 是放到 .env 里面的。但 dotenv 有一个特点，不会覆盖已有的环境变量。也就是说，如果 process.env 上已经有了 HOSTNAME ，那么就不会生效，需要强制将其设为 .env 上的值。

最后，还有一个调试的小技巧。在 node-config 的配置中，任何环境中，只要有 local.EXT(EXT是后缀，可以是 js、yml、json 等) ，那么都会读取，而且会覆盖回去相同字段的配置。即 local.EXT 具有最高优先级。
这个文件一般加入 git 的忽略名单，只有特殊情况下才会用来修改个别配置字段。比如在本地开发的时候，模拟联调环境的配置。

总结

本篇文章解决了文章开始提出的三个问题，其中用到了开源库 dotenv 和 node-config。
通过解决该问题，笔者发现，在解决一个问题的时候，要尽可能考虑使用开源的库。其实像配置的问题，我们内部之前是自己写了方法去按顺序加载文件，并 merge。
但自己写的方法永远只能解决现在遇到的问题，比如之前没有多环境部署需求，只是区分开发和线上，那么在遇到多环境的情况下，就需要对之前的逻辑进行更改。
而使用开源的解决方案，不但能解决现在的问题，还能解决今后遇到的问题(这些问题你现在可能并没有遇到，但对应的库因为使用的人多，已经解决了这些问题)。
so ~ 人生苦短，拥抱开源！以解决问题为导向！才是提高效率的关键！
